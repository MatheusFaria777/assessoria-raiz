# Sessão 2026-06-05 — Progresso e Pendências

## O que foi resolvido nessa sessão

### ✅ Crash de formulários públicos no iOS Safari

**Problema:** GmbForm (`/gmb?c=slug`) e FeedbackPublic (`/nps?c=slug`) crashavam no iOS com "Um problema ocorreu repetidamente" (WebContent process memory crash). O conteúdo aparecia por um segundo, depois o processo morria e o Safari tentava recarregar em loop.

**Três tentativas:**
1. React.lazy() + Suspense para todos os pages — não resolveu (bundle caiu de 340kB para 223kB, crash continuou)
2. Entry points separados no Vite (gmb.html, feedback.html) — não resolveu por si só (bundle caiu para 72kB gzip, crash continuou)
3. Reescrita mobile-first — **resolveu**

**Causas raiz identificadas:**
- `animation: 'raizSlideUp 360ms cubic-bezier...'` em cada step wrapper → GPU compositing layers
- `animation: 'raizFadeIn 260ms ease both'` no BrandPanel com `key={stepKey}` → animava a cada step
- `transition: 'all 160ms ease'` em inputs → stressava GPU em todas as propriedades CSS simultaneamente
- `fontSize: 15` e `15.5` nos inputs → iOS Safari faz zoom automático em inputs < 16px, causando reflow
- `position: sticky; top: 0; height: 100vh; overflow: hidden` no painel → bug clássico iOS scroll/repaint
- `const [isMobile, setIsMobile] = useState(window.innerWidth < 768)` → JS responsivo frágil + resize listener desnecessário

**O que foi feito:**
- `GmbForm.jsx` reescrito do zero — mobile-first, sem animações, `fontSize: 16` em todos os inputs/textareas, `WebkitAppearance: 'none'` e `touchAction: 'manipulation'` em todos os botões, layout via classes CSS
- `FeedbackPublic.jsx` reescrito — removido `isMobile` state + resize listener, layout responsivo 100% CSS via `.pub-layout` / `.pub-panel` / `.pub-mobile-logo`
- `index.css` — removido `@keyframes raizSlideUp` e `@keyframes raizFadeIn`, removido `position: sticky` do `.raiz-gmb-panel`, adicionadas classes `.pub-*`
- `vite.config.js` — multi-entry: `index`, `gmb`, `feedback` como entradas separadas
- `gmb.html` e `feedback.html` criados como HTMLs standalone
- `gmb-main.jsx` e `feedback-main.jsx` criados como entry points separados
- `main.py` — rotas `/gmb` e `/nps` servem HTMLs corretos

**Resultado:** Formulários funcionando no iOS. Deploy confirmado em `sistema.assessoriaraiz.com.br`.

## Diagnóstico iOS (para referência futura)

iOS Safari WebContent tem limite de memória muito mais baixo que desktop. A combinação de animações CSS (GPU layers) + reflow de auto-zoom em inputs < 16px estoura esse limite. Regra: em formulários públicos, nunca usar `transition: 'all'`, sempre `fontSize >= 16` em inputs, evitar animações em elementos de step.
